FS - X86

FS Segment Register
SDT
其实，FS 寄存器用来指示当前线程的 TEB 结构体
然而这种段寄存器只有十六位，如何在 IA-32 系统中32位的指针表示？
实际上，FS段寄存器并非直接指向 TEB 结构体的位置，它持有 SDT 的索引，而该索引持有实际 TEB 地址。SDT 位于内核内存区域，其地址存储在 GDTR (Global Descriptor Table Resiger，全局描述符号表)
由于段寄存器实际存储的是 SDT 的索引，所以它也被称为 “段选择符” (Segment Selector)

FS 保存段选择子 → 通过 GDT/LDT 的段描述符得到 FS 段基址 → Windows 把这个基址设置为当前线程 TEB（或其头部）所在位置 → 于是 FS:[offset] 就能读到 TEB/NT_TIB 的字段。

知乎专栏+1CSDN博客+1

  • FS[0x18] = TEB 起始地址
  • FS[0x30] = PEB 起始地址
  • FS[0] = SEH 起始地址

SEH

SEH 是 Windows 操作系统默认的异常处理机制，在程序源代码中使用 try except finally 关键字来具体实现

SEH 与 Windows 中的 try、catch 具有不同结构！！

OS Exception Handle

同一程序在正常运行时与调试运行时表现出的行为行动是不同的，这是由于 Windows OS 的异常处理方法不同

正常运行时的异常处理方法

  • 进程运行时若发生异常，OS 委托进程进行处理，若进程代码中存在具体的异常处理代码，则能顺利完成，程序继续运行
  • 但如果进行内部没有具体的实现 SEH，OS 就会启动默认的异常处理机制，终止程序运行

调试运行时的异常处理方法

若被调试进程内部发生异常，OS 会首先把异常抛给调试进程处理。

  • 调试器几乎拥有被调试者的所有权限
  • 需要特别指出的是，被调试者内部发生的所有异常（错误）都由调试器处理
  • 所以调试过程中发生的所有异常（错误）都要由调试器管理（被调试者的 SEH 依据优先顺序推给调试器）

被调试者发生异常时，调试器就会暂停运行，必须采取某种措施，完成后继续调试

处理方法如下

1）直接修改异常：代码、寄存器、内存

修改引发异常的位置让其不再异常

2）将异常抛给 被调试者 处理

如果 被调试者 内部存在 SEH（异常处理函数）能够处理异常，那么异常通知会发送给 被调试者，由被调试器自行处理。

如果当前进程在被我们的调试器所调试，我们可以按 SHIFT + F7/F8/F9 命令可以直接将当前异常抛还给 被调试者

3）OS 默认的异常处理机制

若调试器与被调试者都无法处理（或故意不处理）当前异常，则 OS 的默认异常处理机制会触发，终止进程。

Exception

EXCEPTION_ACCESS_VIOLATION (C000 0005)

试图访问不存在或不具有访问权限的内存区域，就会发生 EXCEPTION_ACCESS_VLOLATION（非法访问异常，该异常最常见）。

EXCEPTION_BREAKPOINT (8000 0003)

在运行代码中设置断点后，CPU 尝试执行该地址处的指令时，将会发生 EXCEPTION_BREAKPOINT 异常

INT3

设置该断点的汇编指令为 INT3，对应机器码（IA-32 指令） 0xCC

调试器就是利用该功能实现断点功能

EXCEPTION_ILLGAL_INSTRUCTION (C000 001D)

CPU 遇到无法解析的指令的时候，比如 “0FFF” 指令在 x86 CPU 中未定义，CPU遇到该指令将引发 EXCEPTION_ILLEGAL_INSTRUCTION 异常

EXCETION_INT_DIVIDE_BY_ZERO (C000 0094)

INTEGER（整数）除法运算中，若分母为0（即除0）。

EXCEPTION_SINGLE_STEP (8000 0004)

Single Step（单步）的含义是执行一条指令，然后暂停。

CPU 进入单步模式后，每执行一条指令就会引发 EXCEPTION_SINGLE_STEP 异常，暂停运行。

将 EFLAGS 寄存器的 TF (Trap Flag, 陷阱标志) 位设置为 1 ，CPU就会进入单步工作模式。

SEH 链

SEH 以链的形式存在，第一个异常处理器中若未处理相关异常，就会传递到下个异常处理器，直到得到处理。

从技术层面来看，SEH 是 _EXCEPTION_REGISTRATION_RECORD 结构体组成的链表

Next 成员是指向下一个 _EXCEPTION_REGISTRATION_RECORD 结构体的指针，Handler 成员是异常处理函数（异常处理器）

若 Next 成员的值为 FFFF FFFF，则表示它是链表的最后一个结点，该节点的回调函数是系统设置的一个终结处理函数，所有无人值守的异常都会到达这里。

异常发生时，该异常则会按照 （A）->（B）->（C）的顺序依次传递，直到有异常处理器处理

函数定义如下

  • 异常处理函数接收这4个参数
  • 返回值为 EXCEPTION_DISPOSITION 的枚举类型

_except_handler-> pRecord

首先第一个参数的结构体为 EXCEPTION_RECORD

定义如下，关注两个重要成员

_except_handler-> pContext

其次该结构体中的第三个参数是指向 CONTEXT 结构体的指针

CONTEXT 是用来保存 CPU 寄存器的值，因为多线程环境下需要这样做，当切换去其他线程时，需要保存当前的线程的 CONTEXT 结构体

于是有趣的事情来了，当异常发生时，执行异常代码的线程就会中断运行，转而执行 SEH，OS 就会把线程的 CONTEXT 发送给异常处理函数的相应参数。

该 CONTEXT 结构体的一个成员 EIP（偏移量：B8），注意该成员经常在反调试技术中应用

用法如，在异常处理函数中把 CONTEXT.EIP 设置为其他地址，然后成功返回异常处理函数，这时候就成功执行到了我设置的新 EIP 的位置。

Important Members Of The TEB Structure

TEB 结构体的成员多而复杂，下面是一些比较重要的：

ProcessEnvironmentBlock
首先是Offset 30处的 ProcessEnvironmentBlock 成员，它是指向PEB（Process Environment Block，进程环境块），结构体的指针。

NitTib
TEB 结构体的第一个成员为 NT_TIB 结构体（TIB 是 Thread Infomation Block 的简称，意味线程信息块）
结构体如下：

ExceptionList
• x86 下最常见：指向该线程的异常处理注册记录链表头（传统 SEH 链的入口）。
• x64 下异常处理主要是表驱动机制，这个字段不再扮演 x86 那种“核心链表入口”的角色。

StackBase
• 线程栈的上界（通常是高地址端），用于判断栈范围、分析栈使用情况。

StackLimit
• 线程栈的下界（通常是低地址端），同样用于判断栈范围。

SubSystemTib
• 子系统保留/使用的指针字段；具体用途可能随系统版本/子系统变化，不建议把它当成稳定接口依赖。

FiberData / Version（联合体）
• FiberData：与纤程（Fiber）相关的数据入口（线程被转换成 Fiber 后用于保存/访问 Fiber 的状态/数据）。
• Version：某些定义里用于版本/兼容信息；实现细节可能随版本变化。

ArbitraryUserPointer
• “任意用户指针”：留给运行库/调试器/框架在“线程级别”存放一个自定义指针用的字段（用途不固定）。

Self
• 自引用指针：指向当前 NT_TIB 自己（也等价于指向 TEB 的起始处），可用于快速定位/校验。

比较关心的成员有
• ExceptionList 指向 _EXCEPTION_REGISTRATION_RECORD 结构体组成的链表，就是Windows OS的SEH
• Self 指向 _NT_TIB 结构体自引用指针
◦ 也是 TEB 结构体指针（因为 TEB 结构体的第一个成员就是 _NT_TEB 结构体）

异常处理

例外是程序执行过程中发生的事件，要求在正常控制流之外执行代码。例外分为两种：硬件例外和软件例外。硬件异常由CPU发起。它们可能源于执行某些指令序列，例如除以零或尝试访问无效内存地址。软件异常是由应用程序或作系统显式发起的。例如，系统可以检测何时指定了无效参数值。

结构化异常处理是一种同时处理硬件和软件异常的机制。因此，你的代码会以相同的方式处理硬件和软件异常。结构化异常处理允许您完全控制异常处理，支持调试器，并且适用于所有编程语言和机器。矢量异常处理是结构化异常处理的扩展。

系统还支持终止处理，确保每当执行受保护的代码时，也会执行指定的终止代码块。终端代码无论控制流如何离开被保护的身体都会执行。例如，终止处理程序可以保证即使异常或其他错误发生，在受保护的代码体执行时，清理任务依然会被执行。

异常可以由硬件或软件发起，且可发生在内核模式和用户模式代码中。结构化异常处理为处理内核模式和用户模式异常提供了单一机制。

某些指令序列的执行可能导致硬件发起的异常。例如，当进程尝试从没有适当访问权限的虚拟地址读取或写入时，硬件就会生成访问违规。

需要异常处理的事件也可能发生在软件例程执行过程中（例如，当指定了无效参数值时）。当发生这种情况时，线程可以通过调用 RaiseException 函数显式发起异常。该函数使调用线程能够指定描述异常的信息。

例外可以是连续的或不可连续的。当事件在硬件中不可连续，或延续无意义时，就会出现不可连续异常。不可延续的异常不会终止应用。因此，应用程序可能能够捕获异常并执行。然而，不可连续异常通常是由于堆栈损坏或其他严重问题导致的，这使得从异常中恢复变得困难。

异常调度

当硬件或软件异常发生时，处理器会在异常发生的那一刻停止执行，并将控制权转移给系统。首先，系统会保存当前线程的机器状态和描述异常的信息。系统随后尝试寻找异常处理程序来处理该异常。

异常发生的线程的机器状态被保存在 CONTEXT 结构中。这些信息（称为上下文记录）使系统能够在异常发生点继续执行，前提是异常被成功处理。异常的描述（称为异常记录）保存在EXCEPTION_RECORD结构中。由于它将上下文记录的机器相关信息与异常记录的机器无关信息分开存储，异常处理机制可移植到不同平台。

上下文记录和异常记录中的信息都可以通过 GetExceptionInformation 函数获得，并且可以向因异常而执行的任何异常处理程序开放。例外记录包含以下信息。

  • 一个用于识别异常类型的异常代码。
  • 指示异常是否可延续的标志。任何在非连续异常后继续执行的尝试都会生成另一个异常。
  • 指向另一个例外记录的指针。这有助于在嵌套异常发生时创建异常链表。
  • 例外发生的地址。
  • 一系列参数，提供关于异常的额外信息。

当用户模式代码中出现异常时，系统会使用以下搜索顺序来寻找异常处理程序：

  1. 如果进程正在调试，系统会通知调试器。更多信息请参见调试器异常处理。
  2. 如果进程未被调试，或相关调试器未处理异常，系统会尝试通过搜索异常发生线程的栈帧来寻找基于帧的异常处理程序。系统先搜索当前栈帧，然后按相反顺序搜索前一个栈帧。
  3. 如果找不到基于帧的处理器，或没有基于帧的处理器处理异常，但进程正在调试，系统会再次通知调试器。
  4. 如果进程未被调试，或相关调试器不处理异常，系统会根据异常类型提供默认处理。对于大多数例外情况，默认作是调用ExitProcess函数。

当内核模式代码中出现异常时，系统会搜索内核堆栈的堆栈帧，试图找到异常处理程序。如果找不到处理程序或没有处理程序处理异常，系统会像调用 ExitWindows 函数一样关闭。

调试器异常处理

系统对用户模式异常的处理支持了复杂的调试器。如果异常发生的进程正在调试，系统会生成一个调试事件。如果调试器使用 WaitForDebugEvent 函数，调试事件会使该函数返回并指向 DEBUG_EVENT 结构的指针。该结构包含调试器可用于访问线程上下文记录的进程和线程标识符。该结构还包含一个EXCEPTION_DEBUG_INFO结构，其中包含例外记录的副本。

当系统在寻找异常处理程序时，会尝试两次通知进程的调试器。第一次通知尝试为调试器提供了处理断点或单步异常的机会。这被称为首次通知。用户随后可以发布调试器命令，在执行任何异常处理程序之前作进程的环境。只有当系统找不到处理异常的基于帧的异常处理程序时，才会第二次尝试通知调试器。这被称为最后机会通知。如果调试器在最后机会通知后未处理异常，系统会终止被调试进程。

每次通知尝试时，调试器都会使用ContinueDebugEvent函数将控制权返回系统。在返回控制权之前，调试器可以处理异常并根据需要修改线程状态，或者选择不处理异常。通过 ContinueDebugEvent，调试器可以表示已处理异常，此时机器状态会恢复，线程执行会在异常发生的那个点继续。调试器还可以表示未处理异常，这会导致系统继续寻找异常处理程序。

浮点例外

默认情况下，系统关闭了所有FP例外。因此，计算结果是 NAN 或无穷大，而非例外。在使用结构化异常处理捕捉浮点（FP）异常之前，必须调用_controlfp_s C运行时库函数来开启所有可能的FP异常。要只捕捉特定例外，只需使用对应被捕捉异常的标志。注意，任何处理FP错误的处理程序都应调用_clearfp作为其第一个FP指令。该函数用于清除浮点异常。

异常发生后，大致的流程？。

  1. 如果有调试器：调试器先得到 first-chance 通知
  2. VEH 链：按注册顺序调用所有向量异常处理器（还没开始栈展开）
  3. SEH（栈帧级）：才进入 __try/__except/__finally 这套，必要时开始 unwind（栈展开）
  4. 仍无人处理：走到未处理异常路径（比如 UnhandledExceptionFilter / 用 SetUnhandledExceptionFilter 设置的顶层过滤器） Microsoft Learn+1Microsoft Learn+1

VEH 有“更早、全局”的特点，是微软文档明确写的：在调试器 first-chance 之后、系统开始展开堆栈之前调用。Microsoft Learn

SEH（结构化异常处理） 和 VEH（向量异常处理）和 VCH (向量继续处理程序)

VEH

Windows 提供的“向量异常处理”。你可以通过 API 动态注册回调，例如

AddVectoredExceptionHandler

RemoveVectoredExceptionHandler

Microsoft Learn

矢量异常处理程序是结构化异常处理的扩展。应用程序可以注册一个函数来监视或处理该应用的所有异常。矢量处理程序不是基于帧的，因此你可以添加一个处理程序，无论你在调用帧的哪个位置都会被调用。向量处理程序按添加顺序调用，即调试器收到第一机会通知后，但系统开始解堆栈之前。

要添加矢量继续处理程序，可以使用 AddVectoredContinueHandler 函数。要移除该处理程序，可以使用 RemoveVectoredContinueHandler 函数。

要添加矢量异常处理程序，可以使用 AddVectoredExceptionHandler 函数。要移除该处理程序，可以使用 RemoveVectoredExceptionHandler 函数。

基于帧的异常处理

基于帧的异常处理程序允许你处理异常可能出现在特定代码序列中的情况。基于帧的异常处理程序由以下元素组成。

  • 一套被严密守护的代码体系
  • 滤波表达式
  • 异常处理块

基于帧的异常处理程序以语言特定的语法声明。例如，在Microsoft的C/C++优化编译器中，它们是通过**__try和__except**实现的。更多信息请参见处理者语法。

受保护代码体是一组由过滤表达式和异常处理块提供异常处理保护的一组语句。被保护的主体可以是一个代码块、一组嵌套的块，或是一个完整的过程或函数。使用Microsoft C/C++优化编译器，保护实体被括号（{}）围绕**__try**关键字。

基于帧的异常处理程序的过滤表达式是系统在被保护体内发生异常时评估的一个表达式。该评估导致系统采取以下作之一。

  • 系统停止寻找异常处理程序，恢复机器状态，并在异常发生点继续线程执行。
  • 系统继续寻找异常处理程序。
  • 系统将控制权转移给异常处理器，线程执行在异常处理器所在的栈帧中顺序继续。如果处理程序不在异常发生的堆栈帧中，系统会解开该栈，保持当前栈帧和其他栈帧，直到返回异常处理程序的栈帧。在执行异常处理程序之前，会对因控制权转移给异常处理程序而终止的任何受保护代码体执行终止处理程序。有关终止处理程序的更多信息，请参见终止处理。

滤波表达式可以是简单的表达式，也可以调用一个过滤函数来尝试处理异常。你可以在过滤表达式中调用 GetExceptionCode 和 GetExceptionInformation 函数，获取被过滤异常的信息。GetExceptionCode 返回一个识别异常类型的代码，GetExceptionInformation 返回一个指向包含 CONTEXT 和 EXCEPTION_RECORD 结构的 EXCEPTION_POINTERS 结构的指针。

这些函数无法在滤波函数内部调用，但其返回值可以作为参数传递给滤波函数。GetExceptionCode 可以在异常处理程序块中使用，但 GetExceptionInformation 不能，因为它指向的信息通常在栈中，当控制权转移到异常处理程序时会被销毁。然而，应用程序可以将信息复制到安全存储，以便异常处理程序使用。

过滤函数的优点在于它可以处理异常并返回一个值，使系统从异常发生点继续执行。相比之下，异常处理块的执行是从异常处理程序顺序继续，而不是从异常点开始。

处理异常可能简单到记录错误并设置一个稍后会检查的标志，打印警告或错误消息，或采取其他有限的作。如果可以继续执行，可能还需要通过修改上下文记录来改变机器状态。关于处理页面错误异常的过滤函数示例，请参见“使用虚拟内存函数”。

UnhandledExceptionFilter 函数可以作为过滤器表达式中的过滤函数使用。除非进程正在调试，否则它会返回EXCEPTION_EXECUTE_HANDLER，这种情况下返回EXCEPTION_CONTINUE_SEARCH。

终端处理

终止处理程序确保每当控制流离开某个受保护的代码体时，执行特定的代码块。终端处理程序由以下元素组成。

  • 一套被严密守护的代码体系
  • 当控制流离开被保护机构时执行的终止代码块

终止处理程序以语言特定的语法声明。使用Microsoft C/C++ 优化编译器，它们通过 __try 和 __finally 实现。更多信息请参见处理者语法。

受保护代码体可以是一个代码块、一组嵌套的块，或整个过程或函数。每当被守护的身体被执行时，终止代码块都会被执行。唯一的例外是线程在被保护体执行时终止（例如，当ExitThread或ExitProcess函数从被保护体内部调用时）。

当控制流离开被保护身体时，无论该机构是正常还是异常终止，终止阻断都会被执行。当块中最后一句话句被执行，控制顺序进入终止块时，守护体被视为正常终止。异常终止发生在控制流因异常或控制语句如返回、切换、中断或继续而离开被保护机构时。AbnormalTermination函数可以在终止模块内部调用，以确定被保护的身体是否正常终止。

处理器语法

关键词 描述
__try 开始了一段谨慎的代码体系。与**__except关键字一起使用来构造异常处理程序，或与__finally**关键词一起使用来构建终止处理程序。
__except 开始一段代码，只有在其关联的**__try**块内出现异常时才执行。
__finally 开始一段代码，当控制流离开其关联**__try**块时执行。
__leave 允许立即终止**__try**块，而不会导致异常终止及其性能损失。

编译器还将 GetExceptionCode、GetExceptionInformation 和 AbnormalTermination 函数解释为关键词，若在适当的异常处理语法之外使用它们，则会导致编译器错误。以下是这些功能的简要描述。

功能 描述
GetExceptionCode 返回一个识别异常类型的代码。该函数只能在过滤表达式或异常处理块内调用。
获取例外信息 返回指向包含上下文记录和异常记录指针的EXCEPTION_POINTERS结构的指针。该函数只能在异常处理程序的过滤表达式中调用。
异常终止 表示在执行块中最后一个语句后，控制流是否依次离开了关联的__try块。该函数只能在终止处理程序的**__finally**块内调用。

异常处理者语法

__try和**__except**关键词用于构建基于帧的异常处理程序。以下示例展示了异常处理程序的结构。

注意，__try块和异常处理块都需要大括号（{}）。不允许使用goto语句跳转到**__try**块的正体或异常处理块。该规则适用于异常处理程序和终止处理程序。

__try块包含异常处理程序所保护的受保护代码体。一个函数可以有任意数量的异常处理程序，这些异常处理语句可以嵌套在同一函数中，也可以嵌套在不同函数中。如果**__try**块内发生异常，系统接管控制并开始寻找异常处理程序。有关此搜索的详细描述，请参见异常处理。

异常处理程序只接收单个线程内发生的异常。这意味着如果**__try**块包含对CreateProcess或CreateThread函数的调用，则新进程或线程内发生的异常不会被调派到该处理程序。

系统会评估每个异常处理程序的过滤表达式，保护异常发生的代码，直到异常处理完毕或不再有处理程序。滤波器表达式必须作为以下三个值之一来计算。

价值 含义
EXCEPTION_EXECUTE_HANDLER 系统将控制权转移给异常处理程序，执行在处理程序所在的堆栈帧中继续进行。
EXCEPTION_CONTINUE_SEARCH 系统继续寻找处理者。
EXCEPTION_CONTINUE_EXECUTION 系统停止寻找处理程序，并将控制权返回异常发生的点。如果异常不可连续，则会出现EXCEPTION_NONCONTINUABLE_EXCEPTION异常。

过滤器表达式是在异常处理程序所在函数的上下文中被评估的，尽管异常可能发生在不同的函数中。这意味着过滤器表达式可以访问函数的局部变量。同样，异常处理块也可以访问其所在函数的本地变量。

更多示例请参见“使用异常处理程序”。

有关滤波表达式和滤波函数的更多信息，请参见基于帧的异常处理。

终止处理者语法

__try和**__finally**关键词用于构建终端处理程序。以下示例展示了终止处理程序的结构。

示例请参见“使用终端处理程序”。

与异常处理程序一样，__try块和**__finally块都需要大括号（{}），不允许使用goto**语句跳转到任一块。

__try块包含受终止处理程序保护的代码体。一个函数可以有任意数量的终端处理程序，这些终端处理块可以嵌套在同一函数内或不同函数中。

每当控制流离开**__try块时，执行__finally块。然而，如果您在__try块内调用以下任何函数：ExitProcess、ExitThread 或 abort，__finally**块不会被执行。

__finally块在终止处理程序所在函数的上下文中执行。这意味着**__finally块可以访问该函数的本地变量。__finally**块的执行可以通过以下任一方式终止。

  • 执行该块中最后一条语句并继续到下一条指令
  • 控制语句的使用（返回、中断、继续或 goto)
  • 使用 longjmp 或跳转到异常处理程序

如果**__try块的执行因异常调用了基于帧的异常处理程序的异常处理块而终止，那么__finally块会在异常处理块执行之前被执行。同样，调用 __try 块的 longjmp C 运行时库函数会导致 __finally 块在 longjmp作目标处恢复执行前先执行。如果__try块执行因控制语句（返回、中断、继续或 goto）而终止，则执行__finally**块。

异常终止函数可以在**__finally块内使用，以判断__try块是否顺序终止——即是否达到闭合括号（}）。因调用 longjmp、跳转到异常处理程序，或返回**、中断、继续或 goto 语句而离开**__try**块，被视为异常终止。注意，未能顺序终止会导致系统反向搜索所有堆栈帧，以确定是否需要调用任何终止处理程序。这可能导致性能下降，因为执行了数百条指令。

为避免终止处理程序异常终止，执行应持续到分组结束。你也可以执行**__leave语句。__leave语句允许立即终止__try块，而不会导致异常终止及其性能损失。查看编译器文档，确认__leave**语句是否被支持。

如果**__finally块的执行因返回控制语句而终止，则相当于在封闭函数中切换到**闭括括号。因此，包含函数将返回。

对于CPU级的异常，CPU会通过IDT表寻找异常的处理函数，也就是KiTrapXX例程，会调用CommonDispatchException准备参数，然后调用内核分发函数KiDispatchException进行异常分发。
下面的图是内核异常分发总管KiDispatchException处理的流程。

内核态异常的分发过程：
　　1.如果PreviousMode为KernelMode(0)，那么对于第一轮处理机会，KiDispatchException会试图先通知内核调试器来处理该异常。
　　2.内核变量KiDebugRoutine用来标识内核调试引擎交互的接口函数。当内核调试引擎启用时，KiDebugRoutine指向内核调试引擎KdpTrap，这个函数会进一步把异常信息封装为数据包发送给内核调试器，当调试内核调试引擎没有启用时，KiDebugRoutine指向KdpStub函数，简单处理后返回
　　3.如果KiDebugRoutine返回TRUE，也就是内核引擎处理了异常，那么KiDispatchException便停止继续分发，准备返回。如果KiDebugRoutine返回FALSE，也就是没有处理该异常，那么KiDispatchException会调用RtlDispatchException函数，试图寻找已经注册的结构化异常处理器(SEH)。会遍历异常登记链表，依次执行每个异常处理器。如果某个处理器除了了，RtlDispatchException返回TRUE，否则返回FALSE
　　4.RtlDispatchException返回FALSE，KiDispatchException会试图给内核调试器第二次机会，如果KiDebugRoutine仍然返回FALSE，那么KiDispatchException会认为这是无人处理的异常，会调用KeBugCheckEx
用户态异常的分发过程：
　　1.如果前一模式是用户模式，即PreviousMode参数等于UserMode(1),对于第一次处理机会，KiDispatchException会试图将异常分发给用户态的调试器，如果DebugPort不为空，将异常发送给调试子系统，调试子系统将异常发送给调试器，如果处理了异常分发结束。
　　2.如果调试器没有处理该异常，KiDispatchException修改用户态栈，返回用户层之后执行KiUserExceptionDispatcher，此函数会调用RtlDispatchException来寻找异常处理器，首先遍历VEH，然后遍历SEH，。如果RtlDispatchException返回FALSE，并且当前进程在被调试，那么KiUserExceptionDispatcher会调用ZwRaiseException并将FirstChance设置为FALSE，进行第二轮分发。如果没有被调试，结束进程。
　　3.ZwRaiseException会通过内核服务NtRaiseException把异常传递给KiDispatchException来进行分发。第二次，将异常传递给调试器，如果没有处理将异常分配给ExceptionPort异常端口监听者处理，如果返回FALSE，结束进程。
